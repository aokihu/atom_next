# 0.8 代码评审记录

## 摘要

本次评审聚焦 `0.8.0` 里程碑对应的核心改动，重点从以下角度进行检查：

- 易读性
- 代码阅读布局
- 代码风格一致性
- 代码优雅程度
- 注释完整性
- 逻辑布局
- 模块布局
- 文件布局

总体结论：

- `0.8.0` 已经完成 `FOLLOW_UP` 的最小闭环，功能链路基本完整
- 注释质量和测试覆盖相比旧实现有明显提升
- 但随着 Runtime Context、Intent Request 和 FOLLOW_UP 调度接入，部分文件已经开始出现职责堆叠
- 当前最需要优先处理的是类型契约一致性问题，其次是 `Runtime` 与 `Core` 的类内职责布局

---

## Finding 1

### 标题

`[P1] TaskItem.eventTarget 的运行时值违反了类型契约`

### 位置

- `src/libs/task.ts:79-92`

### 问题描述

当前在构造 `task` 时，把 `eventTarget` 的缺省值设置成了 `null`，但 `RawTaskItem` / `TaskItem` 的类型声明仍然是：

```ts
EventEmitter | undefined
```

随后又通过 `as TaskItem` 的方式把构造结果强转返回。

这会带来两个问题：

1. 类型系统给调用方的预期是“最多为 `undefined`”
2. 运行时真实值却还可能是 `null`

这种“构造时违约、返回时强转”的写法会降低类型可信度，也会让后续维护者难以准确理解边界。

### 影响

- 降低 `TaskItem` 类型定义的可靠性
- 增加后续空值判断分歧
- 让阅读代码时很难判断真实运行时契约

### 建议

优先采用更小改动的方式修正：

- 直接保持 `eventTarget` 缺省值为 `undefined`

备选方案：

- 如果确实需要 `null` 语义，则必须同步调整类型声明为 `EventEmitter | null | undefined`

### 优先级判断

- `P1`

原因：

- 这不是单纯的风格问题，而是明显的类型契约不一致问题

## Finding 2

### 标题

`[P2] Runtime 已经混合了过多职责，阅读入口开始失焦`

### 位置

- `src/core/runtime/runtime.ts:67`

### 问题描述

当前 `Runtime` 已经同时承担了以下职责：

- Runtime Context 维护
- 外部 chat `round` 计算
- `FOLLOW_UP` 上下文维护
- 系统提示词拼装
- Watchman prompt 等待
- Intent Request 日志输出控制
- Intent Request 解析结果整合

从功能上看它能工作，但从代码阅读体验看，`Runtime` 已经不再是一个单纯的“运行时上下文组件”，而变成了一个多职责入口。

这会导致：

- 阅读入口不清晰
- 改动点定位成本上升
- 后续继续扩 milestone 时更容易出现逻辑互相牵连

### 影响

- 降低文件可读性
- 降低职责边界清晰度
- 提高后续维护和重构成本

### 建议

当前阶段不一定要立刻拆文件，但至少应该先在类内做职责收口，建议按三组重新组织：

1. `context`
   - 当前任务同步
   - round 计算
   - followUp 上下文维护

2. `prompt`
   - system prompt
   - user prompt
   - agents prompt
   - intent request prompt

3. `intent-request`
   - safety context
   - rejected / dispatch 日志
   - followUp 提取
   - parseLLMRequest

### 优先级判断

- `P2`

原因：

- 不会立即造成运行错误
- 但已经开始明显影响阅读路径和后续扩展质量

## Finding 3

### 标题

`[P2] 流式输出热路径的职责堆叠过重`

### 位置

- `src/core/core.ts:119-155`

### 问题描述

`onTextDelta` 这段闭包当前同时处理了以下事情：

- 首次进入 `PROCESSING` 状态
- `FOLLOW_UP` 可见前缀注入
- Runtime assistant 输出累计
- `CHAT_CHUNK_APPENDED` 事件发射

单看每一步都不复杂，但全部堆在同一个闭包里，会让最频繁执行的热路径失去层次。

这类代码的问题不在于“看不懂”，而在于：

- 修改时容易顺手把多段逻辑缠在一起
- 新增行为时很容易继续把逻辑塞进闭包里
- 后续排查 chunk 问题时阅读负担偏高

### 影响

- 降低热路径可读性
- 提高局部修改风险
- 不利于后续继续扩展流式行为

### 建议

建议优先做轻量整理，不做大改：

- 提取“生成可见 chunk 文本”的私有方法
- 提取“发送 chunk 事件”的私有方法
- 让 `onTextDelta` 只保留主流程串联

目标不是拆很多代码，而是让热路径结构更清楚。

### 优先级判断

- `P2`

原因：

- 当前功能正确
- 但结构上已经出现继续膨胀的信号

## Finding 4

### 标题

`[P3] 新增测试文件的头部约定不一致`

### 位置

- `tests/core/runtime/runtime.test.ts:1-2`

### 问题描述

当前文件头同时存在：

```ts
//@ts-nockeck
// @ts-nocheck
```

前者是当前项目里的既有约定，后者是 TypeScript 的标准写法。两者并存会带来两个问题：

- 让读代码的人无法判断团队最终规范到底是哪一个
- 让新增测试文件的风格显得不够稳定

### 影响

- 降低测试文件观感一致性
- 增加规范判断歧义

### 建议

统一成单一写法，不要混用。

如果当前项目决定继续沿用既有约定，就只保留项目约定那一条；如果后续想切换到标准写法，也应该统一批量调整，而不是在新文件里混搭。

### 优先级判断

- `P3`

原因：

- 这是风格与规范问题
- 不影响功能正确性

## 综合判断

### 正面评价

`0.8.0` 这批代码有几个明显优点：

- `FOLLOW_UP` 功能链路已经完整串起来
- 提示词拆分成独立 markdown 文件，可维护性提升
- Runtime Context 的设计思路清晰，后续扩展空间明确
- 测试覆盖新增链路较完整
- 注释密度比旧代码明显更高，核心设计意图基本可读

### 主要风险

当前最大的风险不是“功能没实现”，而是：

- 局部职责正在持续聚集
- 类内结构开始变厚
- 类型契约有一处明显不一致

如果继续在这个基础上直接往 `0.9` 叠功能，不先做小规模收口，后面的可读性和维护性会下降得比较快。

---

## 建议处理顺序

建议按照下面的顺序收尾：

1. 先修正 `eventTarget` 的类型契约问题
2. 再整理 `Core.onTextDelta` 的热路径布局
3. 再对 `Runtime` 做类内职责分区
4. 最后统一测试文件头部注释风格

这样做的好处是：

- 优先解决真实契约问题
- 再处理高频执行路径的可读性
- 最后再收口中风险和低风险的结构问题

---

## 结论

`0.8.0` 当前已经具备进入下一阶段开发的基础，但从代码质量角度看，还应该做一轮小规模收尾。

如果只看功能完成度，当前版本是成立的。  
如果看可持续维护性，建议在进入下一里程碑前，先把本次评审中的 `P1 + P2` 项处理掉。
