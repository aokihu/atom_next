Milestone - 0.10
----------------

# 目标

- [ ] 1.在系统内建立独立的 `MemoryService`
- [ ] 2.让 `Runtime(Core)` 具备真实的长期记忆闭环

## 当前背景

从 `0.8` 开始,项目已经为记忆能力预留了协议入口:

- `Intent Request` 中已经存在 `SEARCH_MEMORY`
- `Intent Request` 中已经存在 `SAVE_MEMORY`
- `Runtime Context` 中已经预留了 `<Memory>` 区块

但这些能力目前仍然停留在协议层预留,系统内还没有真正的记忆实现。

因此 `0.10` 的目标不是再增加一个零散功能,而是把之前已经预留好的 Memory 协议真正落地,让长期记忆成为系统内正式存在的基础能力。

---

## 本次里程碑解决的问题

`0.10` 主要解决两个缺口:

### 1.系统内没有真正的长期记忆服务

当前项目虽然已经有 `SEARCH_MEMORY` / `SAVE_MEMORY` / `<Memory>` Context,但实际上还没有:

- 记忆持久化存储
- 记忆搜索
- 记忆读取
- 记忆更新
- 记忆维护
- 记忆关系管理
- 记忆生命周期治理
- 记忆审计追踪

### 2.记忆能力还没有进入当前对话闭环

当前 `Core` 已经具备:

- 安全请求解析
- 安全请求串行消费
- `FOLLOW_UP` 内部任务派生

但 Memory 相关请求还没有真正接入这条链路。

`0.10` 完成后,系统应具备下面这条闭环:

1. LLM 请求搜索或保存记忆
2. 系统真实执行 Memory 动作
3. Memory 结果进入 Runtime Context
4. 下一轮会话或 `FOLLOW_UP` 可以继续消费这些结果

---

## 本次里程碑范围

`0.10` 当前阶段只正式落地 `long` Memory:

- `long`
  - 使用系统内独立 `MemoryService`
  - 承担真正的长期记忆存储与检索

- `short`
  - 当前只保留 Runtime 语义槽位

- `core`
  - 当前只保留 Runtime 语义槽位

也就是说:

- `SEARCH_MEMORY.scope` 为空时,按 `long` 处理
- `SAVE_MEMORY.scope` 为空时,按 `long` 处理

---

## 本次里程碑输出结果

当 `0.10` 完成后,项目应达到下面这个状态:

- `atom_next` 系统内部正式拥有独立的 `MemoryService`
- Memory 连接关系正式采用 `source -> target` 的图边模型
- `SEARCH_MEMORY` / `SAVE_MEMORY` 不再是占位动作
- `Runtime` 的 `<Memory>` 区块开始具备真实内容
- Memory 结果可以进入 `FOLLOW_UP` 链路
- Memory 维护机制开始具备后台运行能力
- Memory 生命周期和审计边界已固定
- 当前系统真正拥有长期记忆能力

## 范围约束

本次里程碑明确不做:

- 不实现 `LOAD_SKILL` 的真实业务
- 不新增新的 `Intent Request` 类型
- 不新增 TUI 的记忆管理界面
- 不复制 `kv-memory` 的 HTTP / MCP / CLI 体系
- 不一次性把 `core` / `short` / `long` 三套记忆全部做完
- 不实现全自动记忆压缩与自动重写等扩展语义
- 不为了 Memory 去重做整个 `Core` 或 `Runtime` 架构
- 不对 LLM 暴露直接删除长期记忆的能力

这里的“不做”不包括保存前的最小去重与冲突判断。  
`0.10` 仍然需要为长期记忆建立最基本的写入治理能力,只是当前阶段不把它扩展成完整的自动整理系统。

---

## 完成标准

当以下条件全部满足时,可以认为 `0.10` 完成:

1. `MemoryService` 已在系统内独立实现
2. 长期记忆存储层已落地
3. `LinkNode` 的有向边模型已固定
4. `SEARCH_MEMORY` 可真实搜索长期记忆
5. `SAVE_MEMORY` 可真实保存长期记忆
6. 保存前最小去重与冲突判断已具备
7. Memory 维护机制已落地
8. `Runtime` 的 `<Memory>` 区块不再是空壳
9. Memory 结果可以进入 `FOLLOW_UP` 链路
10. 相关测试通过
