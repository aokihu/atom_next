# Milestone 0.11 — 基础日志系统与运行观测能力

## 版本目标

`0.11` 的目标是为 `atom_next` 增加一套稳定、可扩展、可替换的基础日志系统。

该里程碑不以增加 Agent 能力为主要目标，而是优先补齐运行时观测能力，使 Core、Runtime、Transport、Bootstrap、Service 启动流程在开发和调试阶段具备清晰的日志输出能力。

本阶段日志系统应作为基础组件存在，而不是作为 `ServiceManager` 管理的业务服务存在。

原因是：日志系统需要在 `ServiceManager` 创建和服务启动之前就可用，尤其是 `bootstrap` 阶段也需要输出调试信息。如果日志系统被设计成 service，则启动早期发生错误时无法通过统一日志系统记录。

---

## 核心设计原则

### 1. 日志系统是基础设施，不是业务服务

日志系统应位于 `libs` 层，而不是 `services` 层。

推荐目录：

```text
src/libs/log/
```

不推荐：

```text
src/services/log/
```

日志系统应在应用启动的最早阶段被创建，并向后传递给 Bootstrap、ServiceManager、Core、API、TUI 等模块使用。

---

### 2. 日志分发与日志处理解耦

日志系统内部可以拆分为：

```text
Logger -> LogHub -> LogSink
```

其中：

```text
Logger:
  业务侧使用的轻量调用入口

LogHub:
  日志事件分发中心，内部可以使用第三方 PubSub 组件实现

LogSink:
  日志处理器，负责保存、显示、写文件或转发
```

这可以避免业务模块直接依赖具体日志库。

业务模块只需要知道：

```ts
logger.info("message");
logger.error("message", { error });
```

不应该知道日志最终写入 console、file、TUI panel 还是远程 collector。

---

### 3. PubSub 只作为日志内部机制，不作为全局事件总线

`0.11` 不引入全局 PubSub。

本阶段的 PubSub 只服务于日志系统内部，用于将日志事件分发给不同 sink。

不建议在 `0.11` 中建立通用事件总线，避免日志事件、业务事件、UI 事件和状态事件混在一起。

推荐命名：

```text
LogHub
```

而不是：

```text
GlobalPubSub
AppEventBus
RuntimeEventBus
```

---

### 4. 业务事件与日志事件必须分离

业务事件用于表达系统状态变化，例如：

```text
chat.enqueued
chat.activated
chat.output.updated
chat.completed
chat.failed
```

日志事件用于表达运行观测信息，例如：

```text
core.runloop.started
workflow.failed
transport.request.started
service.start.failed
bootstrap.config.loaded
```

业务事件不应该被日志系统接管。

日志系统可以记录业务事件的发生，但不能成为业务事件的分发通道。

## 推荐技术选型

### 日志库

推荐使用：

```text
pino
```

原因：

```text
- 性能较好
- 生态成熟
- JSON 日志格式适合后续保存和分析
- 支持扩展 transport
- 适合服务端和 CLI Runtime 项目
```

### 日志内部分发组件

推荐使用：

```text
mitt
```

原因：

```text
- 轻量
- API 简单
- 适合进程内事件分发
- 不会引入复杂事件系统
```

安装命令：

```bash
bun add pino mitt
```

src/libs/log/

```text

  定义 LogLevel、LogEntry、Logger、LogSink 等基础类型

log-system.ts:
  创建日志系统，对外提供 createLogger、emit、subscribe 等高层能力

logger.ts:
  创建业务模块使用的 logger

log-hub.ts:
  日志事件分发中心，内部可以基于 mitt 实现

sinks/pino-sink.ts:
  使用 pino 输出日志

sinks/memory-sink.ts:
  保存最近 N 条日志，供 TUI/API 后续读取
```

## 基础类型设计

```ts
export type LogLevel =
  | "debug"
  | "info"
  | "warn"
  | "error";

export type LogSource =
  | "bootstrap"
  | "service-manager"
  | "core"
  | "queue"
  | "runtime"
  | "transport"
  | "api"
  | "tui"
  | "memory"
  | "watchman"
  | string;

export type LogEntry = {
  id: string;
  time: number;
  level: LogLevel;
  source: LogSource;
  message: string;
  tags?: string[];
  data?: unknown;
  error?: {
    name?: string;
    message: string;
    stack?: string;
  };
};

export type LogOptions = {
  tags?: string[];
  data?: unknown;
  error?: unknown;
};

export type Logger = {
  debug(message: string, options?: LogOptions): void;
  info(message: string, options?: LogOptions): void;
  warn(message: string, options?: LogOptions): void;
  error(message: string, options?: LogOptions): void;
};

export type LogSink = {
  name: string;
  write(entry: LogEntry): void | Promise<void>;
};
```

## 推荐调用方式

### Bootstrap 阶段

```ts
const log = createLogSystem({
  level: "debug",
});

const logger = log.createLogger("bootstrap");

logger.info("Bootstrap started");

const [err, args] = await tryBootstrap();

if (err) {
  logger.error("Bootstrap failed", {
    error: err,
  });

  process.exit(1);
}

logger.info("Bootstrap completed");
```

---

### Core 阶段

```ts
const logger = log.createLogger("core");

logger.info("Core initialized");

logger.debug("Task activated", {
  data: {
    taskId: task.id,
    workflow: taskWorkflow,
  },
});

logger.error("Workflow failed", {
  error,
  data: {
    taskId: task.id,
    sessionId: task.sessionId,
    chatId: task.chatId,
  },
});
```

---

### ServiceManager 阶段

```ts
const serviceManager = new ServiceManager({
  logger: log.createLogger("service-manager"),
});
```

ServiceManager 可以使用日志系统，但日志系统不应该依赖 ServiceManager。

依赖方向必须是：

```text
LogSystem -> independent
ServiceManager -> depends on LogSystem
Service -> depends on LogSystem when needed
Core -> depends on LogSystem
Bootstrap -> depends on LogSystem
```

禁止反向依赖：

```text
LogService -> depends on ServiceManager
```

## 启动流程调整

`0.11` 后启动顺序应调整为：

```text
1. 创建 LogSystem
2. 创建 bootstrap logger
3. 执行 tryBootstrap
4. 创建 RuntimeService / WatchmanService / MemoryService
5. 创建 ServiceManager，并注入 logger
6. 启动 services
7. 创建 Core，并注入 logger 或 log system
8. 创建 API/TUI
9. 启动 Core runloop
```

示意：

```ts
const log = createLogSystem({
  level: "debug",
});

const main = async () => {
  const bootstrapLogger = log.createLogger("bootstrap");

  bootstrapLogger.info("Application starting");

  const [err, args] = await tryBootstrap();

  if (err) {
    bootstrapLogger.error("Bootstrap failed", { error: err });
    process.exit(1);
  }

  const serviceManager = new ServiceManager({
    logger: log.createLogger("service-manager"),
  });

  const core = new Core(serviceManager, {
    logger: log.createLogger("core"),
  });

  core.runloop();
};

main();
```

## 日志等级语义

```text
debug:
  详细调试信息，例如 prompt 导出、task 状态变更、intent request 解析结果

info:
  正常生命周期事件，例如 bootstrap completed、service started、core initialized

warn:
  非致命异常或 fallback，例如 intent prediction fallback、memory search miss

error:
  明确失败，例如 bootstrap failed、workflow failed、transport request failed
```

---

## 日志内容规范

日志 message 应该是稳定、短句、可搜索的英文或固定格式文本。

推荐：

```ts
logger.info("Core runloop started");
logger.error("Workflow failed", { error });
logger.warn("Memory search missed", { data: { query } });
```

不推荐：

```ts
logger.info(`现在开始启动核心了，当前模式是 ${mode}`);
logger.error(`啊出错了 ${error}`);
```

动态内容应放入 `data`，不要拼进 message。

推荐：

```ts
logger.info("Server started", {
  data: {
    host,
    port,
    url,
  },
});
```

不推荐：

```ts
logger.info(`Server started at http://${host}:${port}`);
```

---

## 错误归一化

所有 `unknown` error 都应该被归一化：

```ts
function normalizeError(error: unknown): LogEntry["error"] {
  if (!error) return undefined;

  if (error instanceof Error) {
    return {
      name: error.name,
      message: error.message,
      stack: error.stack,
    };
  }

  return {
    message: String(error),
  };
}
```

业务代码不应该重复写 error 判断逻辑。

## 验收标准

完成 `0.11` 后，应满足：

```text
- bootstrap 阶段出现错误时，可以通过统一日志系统输出
- service 启动阶段可以记录每个 service 的启动结果
- core workflow 失败时可以记录 taskId、sessionId、chatId 和 error
- transport 请求失败时可以记录 provider/model 相关上下文
- 日志系统不依赖 ServiceManager
- 日志系统可以在未来替换 sink，而不影响 Core / Runtime / Bootstrap
- 关键路径不再直接依赖 console.log / console.error
```

---

## 最终结论

`0.11` 的核心不是“做一个日志 service”，而是建立 `atom_next` 的基础观测层。

正确边界是：

```text
libs/log 是基础设施
services 是运行服务
core/runtime 使用 log，但不拥有 log
bootstrap 可以直接使用 log
```

日志系统应优先保证：

```text
早期可用
调用简单
依赖方向正确
后续可替换
不会污染业务事件系统
```
