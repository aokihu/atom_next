# Milestone - 0.9

## 目标

- [x] 1.修正 `0.8.0` 遗留的类型契约问题
- [x] 2.整理 `Runtime` 与 `Core` 的类内职责布局

## 当前背景

`0.8.0` 已经完成了 `FOLLOW_UP` 的最小闭环，实现了：

- Intent Request 基础协议
- Runtime Context 最小可执行版
- Core 内部连续会话

从功能结果看，`0.8.0` 已经具备继续向下开发的基础。  
但是根据 [REVIEW_0.8.md](/Volumes/Projects/atom_next/REVIEW_0.8.md:1) 的评审结果，当前代码还存在几处明显的质量问题，主要集中在：

- 类型契约一致性
- 类内职责堆叠
- 热路径可读性
- 测试文件风格一致性

因此 `0.9` 的目标不是新增功能，而是对 `0.8.0` 的实现做一次小规模质量收尾。

## 目标1

### 修正 Task 类型契约

当前 `TaskItem.eventTarget` 的运行时值与类型声明不一致。

现状：

- `RawTaskItem` / `TaskItem` 的声明是 `EventEmitter | undefined`
- 但任务构造逻辑中实际会把 `eventTarget` 兜底成 `null`
- 最终又通过类型断言返回 `TaskItem`

这会导致：

- 类型系统给出的边界与运行时真实边界不一致
- 调用方无法准确判断该字段的空值语义
- 降低 `TaskItem` 类型本身的可信度

### 本阶段要求

1. 统一 `eventTarget` 的运行时值与类型声明
2. 不允许继续依赖“构造时违约，返回时强转”的方式
3. 优先采用最小改动修正：
   - 让默认值保持为 `undefined`
4. 如果后续确实需要 `null` 语义，则必须同步修改类型声明与调用方判断逻辑

### 验收标准

- `TaskItem.eventTarget` 的运行时值与类型定义一致
- 不再出现依赖错误边界的强转掩盖问题
- 原有 task 相关单元测试继续通过

## 目标2

### 整理 Runtime 的类内职责布局

`Runtime` 在 `0.8.0` 中承担了过多职责，已经从单纯的运行时上下文组件，扩展成一个多职责入口。

当前职责包括：

- Runtime Context 维护
- 外部 chat `round` 计算
- `FOLLOW_UP` 上下文维护
- system prompt / user prompt 输出
- watchman prompt 等待
- Intent Request 日志控制
- Intent Request 结果整合与 follow_up 提取

### 本阶段要求

本阶段不强制拆文件，但必须先做类内职责收口。

建议按下面三组重排私有逻辑：

1. `context`
   - 当前任务同步
   - round 计算
   - followUp 上下文维护

2. `prompt`
   - agents prompt
   - intent request prompt
   - system prompt
   - user prompt

3. `intent-request`
   - safety context
   - rejected / dispatch 日志
   - followUp 提取
   - `parseLLMRequest`

### 目标说明

这里的重点不是“拆更多文件”，而是：

- 让阅读入口更清楚
- 让后续改动点更容易定位
- 避免继续在 `Runtime` 中无序堆功能

### 验收标准

- `Runtime` 内部方法能按职责分组阅读
- 类内结构明显比 `0.8.0` 更清晰
- 不改变当前对外行为
- 原有 Runtime 与 Intent Request 相关测试全部通过

## 目标3

### 整理 Core 流式输出热路径

`Core.onTextDelta` 是当前最频繁执行的路径之一，但在 `0.8.0` 中已经同时承担了多项职责：

- 推进 `PROCESSING` 状态
- 注入 `FOLLOW_UP` 首段可见前缀
- 累计 Runtime assistant 输出
- 发送 `CHAT_CHUNK_APPENDED`

这会让热路径继续扩展时，阅读和维护成本快速升高。

### 本阶段要求

在不改变对外功能的前提下，对流式输出路径做轻量整理：

1. 提取“生成可见 chunk 文本”的私有方法
2. 提取“发送 chunk 事件”的私有方法
3. 保持 `onTextDelta` 只负责主流程串联

### 目标说明

这里不是为了做大规模重构，而是要让下面这条路径更稳定：

- 状态推进
- 文本生成
- 输出累计
- 事件发射

### 验收标准

- `onTextDelta` 主流程比 `0.8.0` 更短、更清晰
- FOLLOW_UP 前缀行为保持不变
- TUI polling 流式显示行为保持不变
- 相关脚本与测试继续通过

## 目标4

### 统一测试文件风格

`0.8.0` 新增测试文件中出现了头部注释风格不一致的问题，例如同时存在：

```ts
//@ts-nockeck
// @ts-nocheck
```

这种混用不会影响功能，但会影响代码观感和规范判断。

### 本阶段要求

1. 测试文件头部注释统一为单一写法
2. 新增与修改的测试文件风格保持一致
3. 不在本阶段额外扩展无关测试规则

### 验收标准

- `tests` 目录下本次涉及的文件头部风格统一
- 不再出现同一文件内混用两种写法的情况

## 实施顺序

`0.9` 建议按下面顺序执行：

1. 先修正 `TaskItem.eventTarget` 的契约问题
2. 再整理 `Core` 的流式输出热路径
3. 再整理 `Runtime` 的类内职责分区
4. 最后统一测试文件风格

这样安排的原因是：

- 先解决真实契约问题
- 再处理高频路径的可读性
- 最后再处理结构整理和风格统一

---

## 范围约束

`0.9` 属于质量收尾里程碑，本阶段明确不做以下内容：

- 不新增新的 Intent Request 类型
- 不扩展新的 Runtime Context 业务能力
- 不改动 FOLLOW_UP 的产品语义
- 不新增新的 TUI 功能
- 不为了“重构而重构”拆分大量文件

本阶段只围绕 `REVIEW_0.8.md` 中已经明确的问题收口。
