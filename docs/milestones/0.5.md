# Milestone - 0.5

## 目标
[ ] 实现TUI交互界面

## 1.TUI交互界面

### 功能说明

当前系统已经具备基础的`server`启动能力以及最小的会话API链路,因此本里程碑的目标是补齐一个可以实际运行的**TUI Client**.  
这个TUI不要求一开始就覆盖完整功能,但是需要具备明确的界面结构、基本的输入输出能力以及与现有`server`的通讯能力,让后续功能能够自然迭代.

TUI在当前阶段更偏向一个**可用的基础交互外壳**,重点是把以下几件事情连通:
- 界面可以显示
- 用户可以输入内容
- 可以把输入发送到当前`server`
- 可以把会话状态和消息结果显示出来

### 技术栈

- 使用`OpenTUI` + `React`实现界面
- 使用`Zustand`管理TUI自身状态
- 使用Bun内置`fetch`与当前`server`通讯

### 开发指南

- TUI相关代码统一放在目录`src/tui`
- 当前阶段优先复用已经存在的HTTP API能力,不额外引入新的通讯协议
- 状态管理以当前界面实际需要为准,避免为了未来功能一次性设计过大的store
- 布局和视觉风格需要有明确方向,但是不要把实现限制成单一写法,允许根据终端特性灵活调整

### 界面布局

- 整体使用三列横向布局,中间区域始终作为主要交互区域
- 左右两列作为信息面板,当前里程碑中以占位、状态信息或少量辅助信息为主
- 中间区域需要同时承担消息展示和用户输入两部分职责
- 消息展示区域应该明显大于输入区域,并支持滚动查看
- 根据终端可用宽度动态显示或隐藏左右面板,但是中间主要交互区域必须始终可用

### 通讯方式

- TUI通过当前已有的`server` API完成会话创建、消息提交和状态获取
- 当前阶段以轮询方式获得chat状态和消息结果即可,不要求在本里程碑中引入更复杂的实时通讯机制
- `tui`模式下直接连接外部`server`, `both`模式下连接当前进程内实际启动成功的`server`

### 色彩主题

- 当前默认使用`Nord`作为主题方向
- 需要保留后续扩展自定义主题的空间
- 主题组织方式可以参考Web常见方案,比如集中定义颜色token或主题配置

### 当前里程碑范围

- 优先完成单轮或连续基础对话所需的最小交互能力
- 左右信息面板不要求在本里程碑中承载复杂业务,保持轻量即可
- 不要求在本里程碑中一次性完成历史会话管理、多窗口布局、复杂快捷键系统或完整主题系统

### 完成指标

1. 可以启动并显示TUI界面
2. 界面具备清晰的三列结构,并且中间交互区域始终可用
3. 左右面板能够根据终端宽度动态显示和隐藏
4. 用户输入功能可用
5. 消息输出功能可用
6. TUI能够通过现有`server`完成至少一条完整的会话交互路径
