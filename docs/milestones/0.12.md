# Milestone 0.12 — Tools 能力与多步工具调用

## 版本目标

`0.12` 的目标是为 `atom_next` 增加一套正式存在的 Tools 能力,让系统在当前 `Queue -> Runtime -> Transport -> Runtime` 主链路上,支持真实工具调用和单轮多步工具交互。

本阶段的重点不是扩展新的业务语义,而是补齐系统在“读取环境、执行工具、基于结果继续推理”这一条链路上的基础能力。

---

## 本里程碑要解决的问题

当前系统虽然已经具备:

- `FOLLOW_UP`
- `Intent Request`
- `MemoryService`
- Runtime Context

但仍然缺少真正的 Tools 闭环。

`0.12` 主要解决下面几个缺口:

1. 系统内没有独立的 Tools 服务边界
2. 正式对话阶段不能真实调用工具
3. 单轮 conversation 不支持多步工具调用
4. 工具执行结果还不能稳定进入后续运行时上下文
5. `FOLLOW_UP` 还不能自然承接工具结果后的连续推理

---

## 本次里程碑输出结果

当 `0.12` 完成后,项目应达到下面这个状态:

- `atom_next` 系统内部正式拥有独立的 `ToolService`
- formal conversation 可以真实调用工具
- 单轮 conversation 支持多步工具调用
- `Runtime` 具备一次性的 `Continuation` 上下文
- `FOLLOW_UP_WITH_TOOLS` 可以承接跨轮继续使用 tools
- 工具权限、预算和执行观测边界已固定

---

## 范围约束

本次里程碑明确不做:

- 不接入 MCP tools 聚合
- 不做工具市场或插件分发体系
- 不做复杂的工具可视化界面
- 不把 `Intent Request` 重构成通用工具协议
- 不复制 `atom` 的整套 agent runner
- 不一次性开放全部高风险工具
