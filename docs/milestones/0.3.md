## 1.Watchman服务

### 功能说明

**watchman**服务是监视并更新系统提示词的服务,在`{workspace}`目录中会存放一些用户自己的提示词文件`AGENTS.md`,这些提示词会被`Runtime(Core)`合并到一起提交给`Transport(Core)`.
出于安全的考量,用户的提示词不能直接合并,需要通过预编译的方式检查用户的提示词文件中是否包含了违反系统安全准则的内容.
编译的过程是通过将用户提示词文件内容上传到**LLM**进行检查,然后从**LLM**返回的安全的用户提示词存入到内存中,当`Runtime(Core)`需要的时候提供使用.
用户可以随时更改提示词文件,因此**watchman**服务将会监视这些提示词文件的变动,并及时的编译与更新内存中的提示词数据

### 开发指南

- 存放在目录`src/services/watchman`
- 该服务会派发一个**worker**用来长期监视文件更改,同时使用定时轮询作为兜底,避免底层文件监听漏事件
- `Runtime(Core)`不会直接读取`watchman`服务,而是只通过`Runtime Service`获取用户提示词和提示词状态
- 当用户提示词尚未准备完成时,`Runtime(Core)`默认行为是等待提示词编译完成,会发送一个警告信息;但是`Runtime(Core)`也可以设置为忽略等待提示词完成

#### 微型LLM服务

`Watchman`服务需要使用LLM来编译提示词,这里需要有一个独立的LLM通讯组件,这个组件与`Transport`没有关系(暂时),使用`AI-SDK`直接提交提示词.

当前实现中,`Watchman`会复用`Runtime Service`对`providerProfiles.basic`的解析能力,自动获得当前`Basic`档位对应的`provider/model`以及其详细配置,然后通过共享的模型工厂创建实际编译器模型.  
也就是说,`Watchman`不会自己重复解析`config`,而是与`Runtime Service`共用这条配置解析链路.

该编译过程是一个简单的提示词编译,因此不需要其他额外的安全校验等,安全措施已经写入到提交的提示词当中.

#### 用户提示词编译后操作
- 在`{workspace}/watchman.json`中记录编译的原信息,方便之后查询用户提示词文件是否更新的时候作为参考
- 对用户的提示词文件做*hash*检查,同样的*hash*提示词文件如果在`watchman.json`找到,则不需要再次编译,否则就需要对用户提示词编译
- 在`{workspace}/compiled_prompts`目录中存放编译后的提示词文件,避免在提示词文件未改动的情况下二次编译
- 编译后的**安全的用户提示词文件**命名用**原始用户提示词文件**的`hash`值命名,比如`{workspace}/compiled_prompts/{origin_prompt_file_hash}.md`

### 对外暴露接口

- `getStatus()` 获取编译用户提示词的状态,主要用于服务侧调试和状态观察
- `getAgentsPrompt()` 获取到`AGENTS.md`编译后安全的提示词内容,以`string`的格式输出
- `recompile()` 强制重新编译


## 2.细化Runtime服务

### 当前问题
- Runtime服务对`config`的执行力度非常粗放,比如获取AI模型,需要后续其他模块自行解析`config`中的参数才能获得

### 改进方案
- 细化Runtime服务对`config`的操作,将`confgi`的不同部分由不同的方法来执行获得,减少其他模块的解析操作

### 与Watchman服务结合

- Watchman服务编译后的提示词文件内容会同步传递给Runtime Service
- Runtime(Core)只允许通过Runtime Service读取用户提示词和用户提示词状态,不再直接依赖Watchman服务
- Runtime Service需要对外暴露4个方法
  - setUserAgentPrompt 设置用户的代理提示词,也就是Watchman中编译后的提示词传给Runtime Service
  - getUserAgentPrompt 获取用的代理提示词,这个功能是暴露给`Runtime(Core)`使用
  - setUserAgentPromptStatus 设置用户的代理提示词状态,这个状态与Watchman必须保持一致,并且状态信息应该为Watchman和Runtime Service共用
  - getUserAgentPromptStatus 获取用户的代理提示词状态,这个功能也是暴露给`Runtime(Core)`使用

### 实际实现说明

- Runtime Service除了保留原有的基础配置访问能力之外,已经细化出按档位读取模型配置的方法,用于统一解析`providerProfiles`
- Watchman在每次状态变化或编译结果变化后,都会立即把最新的用户提示词和状态同步写入Runtime Service
- Runtime(Core)在导出系统提示词时,会通过Runtime Service读取用户提示词状态,决定是等待、忽略还是抛出错误
